# 95306 Phase 3A: Session + Storage Foundation

## Scope

This phase upgrades the verified shipment query PoC into a minimal local collector that:

- reuses the current live `queryCargoSend` path
- writes merged query results into SQLite
- keeps query runs, page summaries, current shipment state, history snapshots, and raw responses
- lets keepalive and query share the same session state files
- supports one-shot CLI runs and an in-process 20 minute scheduler

The current implementation targets local macOS runtime first, while remaining runnable on Windows for development.

### Business rules confirmed in this phase

- `ydid` is the current business primary key.
- `ysfs`
  - `1` = 整车运输
  - `3` = 集装箱运输
- `xh`
  - for container transport, split by `/` and store both the raw string and an array JSON
- `zzl`
  - interpreted as 标载重量
- `zpsj`
  - interpreted as 制票时间
- current stage order:
  - 受理 -> 装车 -> 制票 -> 发车 -> 到站 -> 交付

### Fields kept in JSON instead of hard columns

- the full `queryCargoSend.data.list[i]` record is stored in `shipment_snapshots.record_json`
- current detail subsets are also stored in JSON columns:
  - `detail_json`
  - `loading_unloading_timeline_json`
  - `raw_core_json`
- full merged API response is stored in `raw_api_responses.response_json`

This keeps the schema small while preserving room for future child tables.

## Schema

### `query_runs`

One row per collection execution.

Purpose:
- save query inputs
- record completion status
- record merged counts and return codes
- record whether the query run refreshed session state

Keys:
- primary key: `id`

### `query_run_pages`

One row per fetched page summary.

Purpose:
- persist `page_summaries`
- verify full-page collection

Keys:
- primary key: `id`
- unique key: `(query_run_id, page_num)`

Fields sourced from:
- `send_response.page_summaries[*]`

### `raw_api_responses`

Stores raw API responses for replay and debugging.

Purpose:
- trace merged `queryCargoSend` response
- preserve request payload and response headers

Keys:
- primary key: `id`

Fields sourced from:
- request payload: `ShipmentQueryClient.build_send_payload()`
- response body: `queryCargoSend` merged response
- response headers: live HTTP response headers

### `shipments`

Current-state table. One row per `ydid`.

Purpose:
- store the latest known version of each shipment
- provide the current queryable projection

Keys:
- primary key: `ydid`

Dedup strategy:
- same business shipment is identified by `ydid`

Update strategy:
- every successful collection run upserts the latest projection
- `first_seen_at` is preserved
- `last_seen_at` is updated on every observation

### `shipment_snapshots`

Historical table. One row per shipment per query run.

Purpose:
- preserve historical changes
- keep the full original shipment record

Keys:
- primary key: `id`
- unique key: `(query_run_id, ydid)`

History strategy:
- every collection run inserts a new snapshot for each observed `ydid`

### `stations`

Lightweight station dictionary table.

Purpose:
- persist resolved stations and seen station pairs
- allow future expansion into richer station dimensions

Keys:
- primary key: `tmism`

### `session_states`

Current account session summary mirrored from the ticket files.

Purpose:
- keep the latest session/cookie/token summary in SQLite
- make the collector state visible from one place

Keys:
- primary key: `account_key`

## Session State Manager

Current session state files remain:

- `runtime/95306_ticket_<account>.json`
- `runtime/95306_storage_state_<account>.json`

Current implementation:

- reader:
  - keepalive
  - heartbeat refresh
  - shipment query client
- writer:
  - keepalive via `SessionStateManager.save_bundle()`
  - heartbeat refresh via `SessionStateManager.save_bundle()`
  - shipment query via `SessionStateManager.sync_cookie_jar()`

Query update rule:

- after each live HTTP request, the query client compares the current cookie jar with the saved ticket cookies
- if the cookie jar changed, the latest cookies are written back to both ticket and storage-state files

Concurrency safety:

- writes are guarded by a per-account lock file beside the ticket file
- this is intentionally minimal and file-based for local runtime

## Collection Pipeline

Main entrypoints:

- shared pipeline:
  - `query95306/pipeline.py`
- scheduler:
  - `query95306/scheduler.py`
- CLI:
  - `tools/collect_95306_shipments.py`

Pipeline behavior:

1. Resolve station inputs if keywords are provided.
2. Create a `query_runs` row.
3. Run `queryCargoSend` with automatic pagination.
4. Save page summaries.
5. Save the merged raw response.
6. Upsert current shipments.
7. Insert shipment snapshots.
8. Mirror current session summary into SQLite.
9. Mark the query run completed.

## Automatic Run

Current phase uses an in-process scheduler first.

Reason:
- same code path as manual collection
- no launch agent setup required during schema and pipeline stabilization
- easier to debug while session-writeback behavior is still being verified

Example:

```bash
python3 ./tools/collect_95306_shipments.py \
  --account newts \
  --start-date 2026-03-12 \
  --end-date 2026-03-15 \
  --origin-code 51632 \
  --destination-code 52670 \
  --loop
```

Default interval:

- 1200 seconds
- 20 minutes

Later direction:
- once the query set and local runtime behavior are stable, this can be wrapped by `launchd` on macOS
- Phase 3A does not add `launchd` integration yet

## Known Limits In This Phase

- current raw-response persistence stores the merged `queryCargoSend` response, not every page body separately
- tracking API data is not yet stored as a child node table
- multi-query job configuration is not yet abstracted into a dedicated job file
- session writeback depends on the runtime files as the source of truth; SQLite mirrors summary state but does not replace them

## Future Extension Points

- add `tracking_events` child table from `qeryYdgjNew`
- split `detail_json` into dedicated child tables if downstream use stabilizes
- add per-page raw response rows if replay/debugging needs grow
- add query-job config files for multiple automatic routes
- wrap the scheduler with `launchd` after the pipeline is stable
